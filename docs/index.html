Core conceptsConcetti base

Three ideas explain the whole library. 1) The view pipeline: your data array is never mutated structurally; sort, filter, grouping and tree produce a view (an array of indices) and every API that takes a row index means a view index. 2) Row identity: getRowId gives every row a stable id used by dirty tracking, undo, audit and offline sync — always provide it in real apps. 3) Typed events: every behaviour emits a typed, often cancellable event; wrappers, offline and audit are built only on this public stream, and so can your code. Tre idee spiegano tutta la libreria. 1) La pipeline di vista: l'array data non è mai mutato strutturalmente; sort, filtri, gruppi e tree producono una vista (array di indici) e ogni API che accetta un indice di riga intende l'indice di vista. 2) Identità di riga: getRowId dà a ogni riga un id stabile usato da dirty tracking, undo, audit e sync offline — forniscilo sempre nelle app reali. 3) Eventi tipizzati: ogni comportamento emette un evento tipizzato e spesso cancellabile; wrapper, offline e audit sono costruiti solo su questo flusso pubblico, e può farlo anche il tuo codice.

Editing & validationEditing e validazione

Editing follows Excel exactly: type to overwrite, F2 to edit in place, Enter commits and moves down, Tab moves right, Esc cancels, Ctrl+Enter writes the value into the whole selection, Alt+Enter inserts a line break. Each type gets the right editor: free text, numeric, checkbox toggle for booleans, a native date picker for dates, a dropdown for selects — which becomes a datalist autocomplete beyond 8 options, rejecting out-of-list values. L'editing segue Excel alla lettera: digita per sovrascrivere, F2 modifica sul posto, Invio conferma e scende, Tab va a destra, Esc annulla, Ctrl+Invio scrive il valore su tutta la selezione, Alt+Invio va a capo. Ogni type ha l'editor giusto: testo, numerico, toggle per i booleani, date picker nativo per le date, tendina per le select — che oltre 8 opzioni diventa autocomplete con datalist e rifiuta i valori fuori elenco.

Validation is layered: the column validator rejects bad values, the cell flashes red with the message as tooltip, and the typed validationFailed event lets you surface it anywhere (toast, status bar). La validazione è a strati: il validator di colonna rifiuta i valori errati, la cella lampeggia in rosso col messaggio come tooltip, e l'evento tipizzato validationFailed ti permette di mostrarlo dove vuoi (toast, status bar).

Server-side row modelServer-side row model

For datasets too large for the client. The grid shows all totalCount virtual rows immediately; blocks are fetched as they scroll into view, cached with LRU eviction. Sorting & filtering are delegated: changing them purges the cache and refetches with the current models; a request sequence number discards stale responses, so a fast sort on a slow network can never overwrite fresh data. Not-yet-loaded rows render "…"; loaded rows keep editing, selection, clipboard and dirty tracking. Per dataset troppo grandi per il client. La griglia mostra subito tutte le totalCount righe virtuali; i blocchi sono richiesti quando entrano in vista, con cache LRU. Ordinamento e filtri sono delegati: cambiandoli la cache si svuota e si rifetcha coi modelli correnti; un numero di sequenza scarta le risposte stantie, quindi un sort rapido su rete lenta non può mai sovrascrivere dati freschi. Le righe non arrivate mostrano «…»; quelle caricate mantengono editing, selezione, clipboard e dirty tracking.

By design in server mode: local grouping, insertRows/removeRows and fill across unloaded rows are disabled (the backend owns the dataset); the values filter lists loaded values only.Per design in server mode: raggruppamento locale, insertRows/removeRows e fill su righe non caricate sono disabilitati (il dataset appartiene al backend); il filtro per valori elenca solo i valori caricati.

Offline persistence & sync queuePersistenza offline e coda di sync

Field connectivity comes and goes. attachOfflinePersistence intercepts every user change and persists it immediately to a local queue (localStorage by default, in-memory fallback, injectable adapter for tests/IndexedDB). The queue survives reloads: call restore() at bootstrap to reapply it. flush(sender) is all-or-nothing: if your sender throws (network down), the queue stays intact and you retry. A cell brought back to its initial value leaves the queue by itself; oldValue is the first value seen — what your backend expects for optimistic locking. La rete in cantiere va e viene. attachOfflinePersistence intercetta ogni modifica utente e la persiste subito in una coda locale (localStorage di default, fallback in-memory, adapter iniettabile per test/IndexedDB). La coda sopravvive ai reload: chiama restore() al bootstrap per riapplicarla. flush(sender) è tutto-o-niente: se il sender lancia (rete giù), la coda resta intatta e si ritenta. Una cella riportata al valore iniziale esce dalla coda da sola; oldValue è il primo valore visto — quello che il backend si aspetta per l'optimistic locking.

Performance

Row and column virtualization with element pooling; one rAF-scheduled render pass; analytic offsets (prefix sums + binary search) instead of DOM measurement — that's why detail panels and variable heights never desync. No dependencies, ~39 kB gzip. Practical guidance: provide getRowId; keep valueFormatter/cellClass pure and fast (they run on visible cells only); prefer applyChanges for bulk writes; beyond ~1M rows or heavy server data, switch to server-side mode. Zero dipendenze, ~39 kB gzip. In pratica: fornisci getRowId; tieni valueFormatter/cellClass puri e veloci (girano solo sulle celle visibili); preferisci applyChanges per scritture massive; oltre ~1M righe o con dati pesanti lato server, passa alla modalità server-side.

Documented limitsLimiti documentati

Deliberate semantics you should know — written down so nobody discovers them in production. Formulas: references are positional on data coordinates; sorting never changes a formula's meaning, and inserting/removing rows recalculates but does not rewrite references (no Excel-style relative adjustment). Server mode: no local grouping, row insert/remove or fill across unloaded rows; the values filter lists loaded values only. Tree mode: no insert/remove/groupBy. XLSX: values + headers of a single sheet (no styles/formulas/multi-sheet). Nested detail grids scroll within their panel when their own children expand. Semantiche scelte apposta, da conoscere — scritte qui perché nessuno le scopra in produzione. Formule: riferimenti posizionali sulle coordinate dei dati; l'ordinamento non cambia mai il significato e inserire/eliminare righe ricalcola ma non riscrive i riferimenti (niente aggiustamento relativo stile Excel). Server mode: niente raggruppamento locale, insert/remove righe o fill su righe non caricate; il filtro per valori elenca solo i valori caricati. Tree mode: niente insert/remove/groupBy. XLSX: valori + intestazioni di un foglio (niente stili/formule/multi-foglio). Le griglie detail annidate scrollano nel loro pannello quando i loro figli si espandono.
